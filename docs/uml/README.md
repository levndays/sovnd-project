# UML-діаграми СОВНД (PlantUML)

Чотири діаграми за моделлю «4+1» (Philippe Kruchten), що відповідають
рисункам §3 курсової роботи.

| Файл | Розділ | Тип діаграми (UML 2.5) | Що відображає |
| :---- | :---- | :---- | :---- |
| `fig-3-1-logical.puml`    | §3.2 | Component diagram   | 12 компонентів СОВНД, провайдені/потрібні інтерфейси, шлях події |
| `fig-3-2-sequence.puml`   | §3.3 | Sequence diagram    | Реалізована модель з двома потоками (T_poll, T_main) + окремим процесом API |
| `fig-3-3-deployment.puml` | §3.4 | Deployment diagram  | Розгортання двох Docker-контейнерів на Linux-хості з eBPF-зондом у ядрі |
| `fig-3-4-usecase.puml`    | §3.6 | Use case diagram    | 4 прецеденти + 4 актори з коректними `<<include>>` / `<<extend>>` |

## Рендеринг

Локально:

```bash
sudo pacman -S plantuml        # Arch
sudo apt install plantuml      # Debian/Ubuntu

# Окрема діаграма → PNG:
plantuml docs/uml/fig-3-1-logical.puml

# Усі чотири → SVG (масштабована якість, потрібно для друку):
plantuml -tsvg docs/uml/*.puml
```

Онлайн (без встановлення): <https://www.plantuml.com/plantuml/uml/>

## Дотримання UML 2.5 — критичні моменти

Нижче — інваріанти, на яких сувориться більшість академічних рецензентів.
Кожен файл явно їх дотримує.

### Fig 3.1 — Component diagram

- `skinparam componentStyle uml2` — використано канонічну UML 2.5 іконку
  компонента (прямокутник із двома малими прямокутниками).
- Інтерфейси оголошено як `interface` із подальшим символьним зв'язком
  `--` для provided та `)--` для required (лолі-поп / розетка).
- Стрілки залежності — пунктирні `..>` (UML «dependency»),
  на відміну від суцільних (асоціація / агрегація).

### Fig 3.2 — Sequence diagram

- `skinparam style strictuml` — трикутні наконечники замість стрілкових,
  як вимагає UML 2.5 §17.4.4.
- Синхронні повідомлення: `->` (повний наконечник);
  асинхронні: `->>` (відкритий);
  повернення: `-->` (пунктир).
- Активаційні смуги `activate`/`deactivate` для кожного виклику.
- Комбіновані фрагменти `alt`/`loop` для умовної логіки та повторень.
- `hide footbox` — приховано дублікат box-ів унизу, як прийнято для
  компактного академічного формату.

### Fig 3.3 — Deployment diagram

- Стереотипи стандартні: `«device»`, `«executionEnvironment»`,
  `«container»` (UML 2.5 §19.3).
- Артефакти оголошено через `artifact` (а не `component`) — це принципова
  різниця: артефакт є фізично-розгорнутою сутністю.
- Шляхи комунікації — пунктирні залежності зі стереотипами протоколів
  (`«RingBuffer»`, `«DockerSDK»`, `«HTTP/WebSocket»`).
- Чітка ієрархія вкладеності: Linux Host → kernel/Docker Engine →
  контейнери → артефакти.

### Fig 3.4 — Use case diagram

- **Напрямки `<<include>>` та `<<extend>>` найчастіше плутають.**
  Тут вони дотримані строго за UML 2.5 §16.3:
  - `UC2 ..> UCres : <<include>>` — базовий UC `UC2` ЗАВЖДИ викликає
    включений `UCres`; стрілка вказує НА включений UC.
  - `UC3 ..> UC1 : <<extend>>` — розширювальний `UC3` УМОВНО додається
    у точці розширення базового `UC1`; стрілка вказує НА базовий UC.
- Стереотип `<<system>>` на акторах `Kernel` та `Docker` позначає
  системних (не людських) акторів — також за UML 2.5 §16.3.
- Межа системи — `rectangle` (а не `package`), щоб коректно
  візуалізувати «System Boundary».
- Точки розширення (`extension point`) явно зазначено у мітках
  стрілок `<<extend>>`.

## Зв'язок з кодом

Кожен компонент / артефакт у діаграмах посилається на конкретний файл
репозиторію (у нотатках). Це робить діаграми відтворюваними і
прив'язує дизайн до реалізації — те, на чому професори зазвичай
наполягають у курсових з програмної інженерії.
